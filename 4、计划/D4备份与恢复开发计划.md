# D4 备份与恢复开发计划

本计划针对 M1/D4 阶段的"备份与恢复"工作，目标是实现管理器内的"备份与恢复"闭环机制，支持手动备份、列表查看以及通过备份 ID 原子回滚恢复本地代码和注册表快照。

## 涉及文件

- `src/backup/create.ts` - 实现手动备份 `createManualBackup` 与扫描获取备份列表 `listBackups`
- `src/backup/restore.ts` - 实现备份还原 `restoreBackup`
- `src/backup/cleanup.ts` - 备份清理策略占位常量（本阶段不实现清理函数）
- `src/cli/backup.ts` - 实现 `asm backup` 创建/列表命令行
- `src/cli/restore.ts` - 实现 `asm restore <id>` 还原命令行
- `src/server/routes/backup.ts` - 路由适配 API 列表与创建
- `src/server/routes/restore.ts` - 路由适配 API 还原触发

## 1. 拟实施工作项

### 1.1 手动快照备份与列表拉取
- 在 `src/backup/create.ts` 中：
  - 实现 `createManualBackup(root, skillName?, reason?)`：
    - 生成 `bk_<timestamp>_<uuid8>` 格式的 `backupId`。
    - 将 `library/registry.json` 保存到 `registry-snapshot.json` 快照中。
    - 若指定 `skillName`：仅复制并备份 `library/skills/<skill-name>` 目录。
    - 若未指定 `skillName`：复制并备份整个 `library/skills` 目录。
    - 写出 `index.json` 索引文件。
  - 实现 `listBackups(root)`：
    - 遍历 `config.backupDir` 中所有包含 `index.json` 的 `bk_*` 目录。
    - 解析索引元数据，按创建时间倒序（降序）返回数组。
    - 当 `config.backupDir` 不存在时返回 `[]`，不抛错。

### 1.2 倒序原子恢复机制
- 在 `src/backup/restore.ts` 中实现 `restoreBackup(backupId, root)`：
  - 根据 `backupId` 载入备份 `index.json`，找不到则抛 `BACKUP_NOT_FOUND`。
  - 倒序（Reverse）遍历备份项目列表（优先恢复文件，最后恢复 registry 注册表文件，保障原子事务安全性）。
  - 对 Skill 目录还原前，先物理彻底清空（删除）本地当前目录以防残留，再复制还原备份副本。
  - 对 `rule` 类型 item 仅做文件级恢复；不存在则删除目标文件。**该路径当前仅作恢复端占位，规则文件的备份来源属于 D8 范围。**

### 1.3 命令行接口对接
- 扩展 `src/cli/backup.ts`：
  - 支持 `asm backup` 创建全局手动备份。
  - 支持 `-s, --skill <name>` 参数备份特定技能。
  - 支持 `-l, --list` 表格化展示所有可用备份的详细列表。
  - 支持 `--reason <text>` 自定义原因，默认 `Manual backup`。
- 扩展 `src/cli/restore.ts`：
  - 支持 `asm restore <backup-id>` 触发还原流程并打印详情。

### 1.4 API 路由更新
- `GET /api/backups` 返回 `{ backups: BackupIndex[] }`，按 `createdAt` 倒序。
- `POST /api/backups` 创建备份：body `{ skillName?: string; reason?: string }`，返回 `BackupIndex`。
- `POST /api/restore` body `{ backupId: string }`，返回 `{ success: true, index: BackupIndex }`。
- 错误响应统一走 D5 阶段引入的全局错误中间件，错误体形如 `{ error: { code, message, details? } }`。

### 1.5 BackupItem 字段约束
- D4 阶段 `BackupItem.type` 固定为 `'registry' | 'skill' | 'rule'`。
- D4 阶段 `createManualBackup` 不写 `targetType` / `targetAgent` / `targetSkillPath`（这些字段由 D3a 的 `backupBeforeSync` 与 D3b 的 `backupLocalLibraryBeforePull` 写入）。`restore.ts` 必须兼容两种索引格式：旧版（无 targetType）按路径推断，新版按字段读取。
- `BackupItem.skillName` 在 D4 中可省略；省略时表示"备份了整个 `library/skills` 目录"。

### 1.6 清理策略（仅占位常量）
- `src/backup/cleanup.ts` 当前只导出 `defaultBackupRetentionPolicy = { keepLatest: 50, keepDays: 30, maxBytes: 2 GiB }`。
- **D4 不实现 `cleanupByPolicy` 等实际清理函数**。后续 D5 / D4.x 阶段补清理 CLI 命令 `asm backup cleanup [--dry-run]` 与 `POST /api/backups/cleanup`。

### 1.7 调用点与触发语义
- D4 仅暴露手动备份与手动恢复 API。
- 备份触发点（覆盖前自动备份）由其他阶段提供：
  - D1 `importSkill --force` 调用 `backupSkillAndRegistry`（已在 `src/backup/create.ts` 实现）。
  - D3a `applySyncPlan` 在 modify 路径调用 `backupBeforeSync`（在 `src/sync/backup-user.ts` 实现）。
  - D3b 反向拉取本地库前调用 `backupLocalLibraryBeforePull`（待 D3b 实施）。
- D4 不修改上述自动备份调用点的索引格式。

## 2. 验证机制

### 自动化测试

`tests/integration/backup.test.ts` 已覆盖：

- 全局备份 + 单 Skill 备份 + 列表倒序 + 还原（registry + skill 内容回滚 + 残留文件清理）+ 不存在 ID 抛 `BACKUP_NOT_FOUND`。

D4 阶段建议补：

- `registry.json` 不存在时仍能成功创建 backup（仅 skill 项，没有 registry 项）。
- rule 类型的 BackupItem 能被 restore 还原到正确路径。
- restore 中途失败时不留下半写入状态（备份目录保留，便于人工排查）。

执行：
- `pnpm run typecheck`
- `pnpm run test`

### 手动检查

- 运行 `pnpm dev backup` 执行手工备份。
- 运行 `pnpm dev backup --list` 确认列表输出。
- 运行 `pnpm dev backup --skill <name> --reason "test"` 确认指定 Skill 备份。
- 运行 `pnpm dev restore <id>` 验证还原恢复动作。
- Web UI `Backups` 页面点 `Restore` 后确认二次确认窗口。

## 3. 风险与待确认

- **索引格式向前兼容**：D4 BackupItem 不写 `targetType` 是为了保持 D4 计划简单，但 D3a / D3b 备份会写入该字段。`restore.ts` 必须兼容两种格式，否则 D3a 生成的备份无法被 D4 还原命令恢复。建议 D5 阶段统一索引格式（必填 `targetType`）。
- **清理策略入口缺位**：D4 仅暴露占位常量，没有 `asm backup cleanup` 命令。备份目录可能膨胀，依赖用户手动删除。建议 D4.x 阶段补一个最小清理命令（仅 `--keep-latest` 一个参数即可）。
- **rule 备份来源缺失**：D4 实现了 rule 恢复，但 D4 范围内没有创建 rule 备份的入口（属于 D8）。当前若 `restore` 遇到 rule item 且 `backupPath` 不存在，会删除目标 rule 文件，存在误删风险。D8 实施前建议在 `restore.ts:34-40` 增加"目标 rule 文件不存在时跳过、不删除"的容错。
- **`asm restore` 无 `--dry-run`**：当前命令直接物理覆写，不可撤销。Web UI 已经做了二次确认，但 CLI 没有。建议 D4.x 增加 `--dry-run` 显示将影响的文件清单。
- **备份目录跨工作区**：当前 `backupDir` 是相对路径；如果用户切换 workspace root，备份可能被孤立。CLI/server 应在启动时校验 `backupDir` 是否落在当前 workspace root 下，跨 workspace 的备份应明确拒绝（待 D5 全局错误中间件落地后由路径校验统一处理）。
