# Agent 协作规范

本文档定义本仓库内多 Agent 协作的统一工作方式。所有角色 Agent 在开始任务前，先阅读本规范，再阅读自己的角色文件。

## 1. 协作目标

围绕 Skill Manager 首版建设，保证需求、架构、计划、代码、测试之间可追踪、可交接、可验收。

核心原则：

- 以用户确认的需求和当前仓库状态为准，不凭空扩展范围。
- 先确认影响面，再修改文件。
- 文档、代码、测试三者保持一致。
- 每次交接必须说明已完成、未完成、验证结果和风险。
- 对高风险写入逻辑坚持 plan/apply、dry-run、备份和可恢复。

## 2. 角色分工

| 角色 | 主责 | 主要输入 | 主要输出 |
| --- | --- | --- | --- |
| Planner Agent | 拆分需求、排优先级、维护开发计划 | `1、需求`、用户新要求、当前实现状态 | `4、计划` 下的任务拆分、里程碑、验收口径 |
| Architect Agent | 维护架构边界、接口契约和关键技术决策 | `1、需求`、`4、计划`、现有代码结构 | `2、架构`、`3、详细设计` 的架构约束和设计说明 |
| Developer Agent | 实现功能、修复缺陷、维护开发记录 | 已确认任务、架构约束、现有测试 | `src`、`web`、`library`、`5、开发` 中的实现和记录 |
| Tester Agent | 设计测试、执行验证、输出风险结论 | 需求验收标准、实现改动、已知风险 | `tests`、测试报告、缺陷清单、验收结论 |

## 3. 文档归属

| 内容类型 | 归属位置 |
| --- | --- |
| 原始需求、范围、验收目标 | `1、需求/` |
| 架构总览、技术决策、模块边界 | `2、架构/` |
| 目录、接口、数据结构、流程细节 | `3、详细设计/` |
| 任务拆分、阶段计划、优先级 | `4、计划/` |
| 开发记录、问题记录、变更说明 | `5、开发/` |
| Agent 角色与协作规范 | `.agents/` |
| 自动化测试与集成验证 | `tests/` |

规则：

- 不在代码注释里沉淀需求背景，需求背景写入文档。
- 不在计划文档里记录最终实现细节，最终实现差异写入开发记录或详细设计。
- 架构变化必须同步到 `2、架构` 或 `3、详细设计`。
- 任务拆分变化必须同步到 `4、计划`。

## 4. 标准协作流程

### 4.1 需求进入

Planner Agent 负责把需求拆成可执行任务：

1. 从用户描述和 `1、需求` 中提取目标。
2. 检查当前代码与已有计划，避免重复拆分。
3. 标注范围：首版必做、首版暂缓、后续扩展。
4. 为任务写清楚输入、输出、涉及文件、验收标准。

交付要求：

- 每个任务能由 Developer Agent 独立开始。
- 每个任务有明确完成标准。
- 不把未确认的新能力混入当前任务。

### 4.2 架构确认

Architect Agent 负责在开发前确认结构和边界：

1. 判断任务是否影响现有模块边界。
2. 明确核心接口、数据模型、配置来源和错误处理。
3. 对高风险文件写入、路径安全、备份恢复给出约束。
4. 必要时更新架构或详细设计文档。

交付要求：

- 说明哪些模块可以改，哪些模块不应改。
- 说明是否需要迁移、兼容或回滚策略。
- 对不确定点给出待确认项，而不是直接默认实现。

### 4.3 开发实现

Developer Agent 负责按已确认任务实现：

1. 开始前检查 `git status`，识别已有未提交变更。
2. 阅读相关需求、计划、架构和代码。
3. 保持改动范围与任务一致，不做无关重构。
4. 优先复用现有类型、模块和工具函数。
5. 修改写文件逻辑时必须保留 dry-run、备份、冲突检测或路径保护。
6. 完成后运行与改动匹配的检查。

交付要求：

- 说明改了哪些文件和为什么改。
- 说明已运行哪些验证。
- 说明未运行的验证和原因。
- 对遗留风险给出下一步处理建议。

### 4.4 测试验收

Tester Agent 负责验证结果是否满足需求：

1. 从需求和计划中抽取验收点。
2. 补齐单元测试、集成测试或手工验证步骤。
3. 优先覆盖文件写入、路径越界、冲突、备份恢复、dry-run。
4. 输出通过项、失败项、阻塞项和剩余风险。

交付要求：

- 测试结论必须绑定具体命令或操作。
- 失败项必须包含复现路径。
- 不能只给“通过”结论，必须说明覆盖范围。

## 5. 交接格式

Agent 之间交接时使用以下格式：

```md
## 交接摘要

- 当前任务：
- 已完成：
- 未完成：
- 修改文件：
- 验证结果：
- 风险与阻塞：
- 建议下一步：
```

交接必须做到：

- 文件路径明确。
- 未完成项具体。
- 验证结论可复现。
- 不隐藏失败和不确定点。

## 6. 变更约束

通用约束：

- 不覆盖用户或其他 Agent 的未确认改动。
- 不删除无关文件。
- 不把文档重排、格式化和功能改动混在一起。
- 不引入新依赖，除非任务明确需要且说明理由。
- 不把 Web UI、CLI、server 各自实现一套写入逻辑；写逻辑必须沉到核心模块。

文件写入约束：

- 写入 Agent 目录或项目目录前必须先生成 plan。
- apply 必须基于 planId 或等价的已确认计划。
- 覆盖前必须备份。
- 目标端存在非本工具管理的同名内容时默认冲突，不自动覆盖。
- 项目级写入必须校验真实路径落在注册项目内部。

## 7. 任务状态

任务状态使用以下口径：

| 状态 | 含义 |
| --- | --- |
| Todo | 已拆分，尚未开始 |
| Doing | 正在实现或验证 |
| Blocked | 遇到明确阻塞，无法继续 |
| Review | 实现完成，等待测试或人工确认 |
| Done | 验收通过，文档和测试已同步 |

Blocked 必须写明：

- 阻塞原因。
- 已尝试的处理方式。
- 需要用户或外部环境提供什么。

## 8. 验证门槛

不同改动对应不同验证：

| 改动类型 | 最低验证 |
| --- | --- |
| 纯文档 | 检查路径、内容结构、链接是否准确 |
| 类型或纯函数 | `pnpm run typecheck` + 相关单元测试 |
| CLI 行为 | CLI 命令实跑 + 集成测试 |
| server API | API 路由测试或本地请求验证 |
| Web UI | typecheck + 页面手工验证或 Playwright |
| 文件写入逻辑 | 临时目录集成测试 + 备份/恢复/冲突场景 |

如果无法运行验证，必须说明原因，不得把未验证内容描述为已验证。

## 9. 冲突处理

当多个 Agent 或用户同时改动同一文件：

1. 先停止继续写入该文件。
2. 阅读当前文件内容和 `git diff`。
3. 判断冲突是否影响当前任务。
4. 能合并则保留双方意图并说明。
5. 不能判断则交回用户确认。

禁止：

- 用重置、覆盖、删除来“清理”不属于当前任务的改动。
- 在不了解改动来源时恢复旧版本。

## 10. 首版协作优先级

当前首版开发优先级：

1. D0 基础设施补齐。
2. D1 Skill 导入与注册表。
3. D2 用户级 scan/list/diff。
4. D3 用户级 sync plan/apply/backup/tag。
5. D4 备份恢复。
6. D5 server API。
7. D6 Web UI。
8. D7 项目工作区与项目级注入。
9. D8 规则模板同步。
10. D9 Watch Mode 与 Antigravity。

开发开始时，默认先推进 D0-D3，除非用户明确调整优先级。
