# D2 用户级扫描、列表与差异开发计划

本计划针对 M1/D2 阶段的"用户级扫描、列表与差异"工作，目标是实现本地注册技能列出、多端（本地与 Agent 目标）扫描比对与状态识别、以及文本/文件级差异对比功能。

## 涉及文件

- `src/adapters/types.ts` - 在 `Adapter` 接口上新增 `scanUserSkills()`，新增 `TargetSkillInfo` 类型
- `src/adapters/base.ts` - 在 `BaseAdapter` 中实现通用的 `scanUserSkills` 逻辑
- `src/adapters/scan.ts` - 适配器扫描入口与状态判定工具
- `src/sync/deploy-tag.ts` - 部署标记读取 `readDeployTag`（与 D3a 共享）
- `src/sync/multidirection.ts` - 多端状态归一化（与 D3a 共享 `identifySkillState`）
- `src/rules/diff.ts` - 扩展：在现有 `diffText` 基础上补 `diffDirectories` 目录级文件差异
- `src/cli/list.ts` - 实现 `asm list` 技能列出命令
- `src/cli/scan.ts` - 实现 `asm scan` 多端扫描比对状态命令
- `src/cli/diff.ts` - 实现 `asm diff <skill-name> <target-key>` 文本级差异比对命令
- `src/server/routes/skills.ts` - 列表 / 扫描接口返回真实状态
- `src/server/routes/scan.ts` - 触发扫描并返回归一化状态
- `src/server/routes/diff.ts` - 接受 `skillName + targetKey`，返回文件级 + 文本级 diff

## 1. 拟实施工作项

### 1.1 类型定义扩展

- 在 `src/adapters/types.ts` 中新增 `TargetSkillInfo`：
  ```ts
  interface TargetSkillInfo {
    name: string
    localPath: string
    checksum: `sha256:${string}`
    deployTag?: DeployTag      // 若存在 .skill-manager-deploy.json
    detected: boolean           // 目标根目录存在
  }
  ```
- 扩展 `Adapter` 接口：
  ```ts
  interface Adapter {
    readonly agent: AgentId
    detect(): Promise<boolean>
    getTargetPaths(): AdapterTargetPaths
    scanUserSkills(): Promise<Record<string, TargetSkillInfo>>
  }
  ```

### 1.2 适配器扫描实现

- 在 `src/adapters/base.ts` 中提供 `scanUserSkills` 默认实现：
  - 扫描 `userSkillPath` 下所有含 `SKILL.md` 的直接子文件夹；
  - 对每个子目录调用 `checksumDirectory`；
  - 读取 `.skill-manager-deploy.json`（通过 `readDeployTag`）填充 `deployTag`；
  - 目标根目录不存在时返回空对象，**不抛错**。
- `src/adapters/scan.ts` 提供状态归一化工具 `identifySkillState(source, target)`，输出：
  - `identical`：目标存在、checksum 一致；
  - `missing`：目标不存在；
  - `changed`：目标存在、checksum 不一致、目标含 deploy tag；
  - `conflict`：目标存在、checksum 不一致、目标无 deploy tag；
  - `untracked`：目标存在、但 registry 中无对应 Skill（仅作扫描结果展示，不参与 plan）。
- Gemini/Antigravity 适配器在 D3a 不写入，但 D2 必须支持扫描与展示，状态用同样的五元组。

### 1.3 目录文本 diff 比对

- 在 `src/rules/diff.ts` 中补：
  - `diffText(fromName, toName, before, after)`（已存在）；
  - `diffDirectories(sourceDir, targetDir)`：
    - 收集两端文件的并集（过滤 `.` 开头文件、忽略 `node_modules` 等）；
    - 对每个相对路径：
      - 仅 source 存在 → 标记 added；
      - 仅 target 存在 → 标记 removed；
      - 两侧都存在 → 用 `createTwoFilesPatch` 生成文本级 patch。
    - 返回 `{ files: Array<{ path: string; status: 'added'|'removed'|'changed'; patch?: string }> }`。

### 1.4 CLI 命令设计与集成

- `asm list`：读 registry，按以下字段表格化输出：
  - name / version / description
  - canonical localPath（`library/skills/<name>`）
  - dev localPath（registry `localPath` 字段，溯源用）
  - checksum（前 12 位）
  - `syncedTargets`
- `asm scan`：调用 `createAdapters(config)` 中所有 `enabled` 适配器的 `scanUserSkills`，与 registry 比对：
  - 对每个 Skill × 每个 Target 给出 `identical/missing/changed/conflict`；
  - 输出 `untracked` 列表（目标端存在但 registry 缺失的 Skill）；
  - 目标根目录不存在时打印"未检测到 <agent> 用户级 Skill 目录"，**不抛错**。
- `asm diff <skill-name> <target-key>`：
  - target-key 形如 `claude:user`、`codex:user`；
  - source 取 `library/skills/<skill-name>`；
  - 调 `diffDirectories` 输出文件级清单，对 changed 的文本文件再调 `diffText` 打印 patch。
  - 当目标不存在或 registry 不存在 Skill 时，给出明确错误并退出码非零。

### 1.5 server API 接入

- `GET /api/skills` 返回 `{ skills: SkillState[], lastScan?: ScanResult }`。
- `POST /api/scan` 返回 `{ results: Record<TargetKey, Record<skillName, 'identical'|'missing'|'changed'|'conflict'>>, untracked: Record<TargetKey, string[]> }`。
- `GET /api/diff?skill=<name>&target=<key>` 返回 `{ files: DiffFileEntry[] }`。

## 2. 状态机统一

为 D2 / D3a 共用，状态机统一为：

| 状态 | 触发条件 | 是否可 apply |
| --- | --- | --- |
| identical | source 与 target checksum 相同 | 否（默认 skip） |
| missing | target 不存在 | 是（create） |
| changed | target 存在、checksum 不同、target 有 deploy tag | 仅当命令带 `--allow-managed-modify` |
| conflict | target 存在、checksum 不同、target 无 deploy tag | 否（默认拒绝） |
| untracked | target 存在、registry 不存在该 Skill | 仅展示，不参与 plan |

注意：`untracked` 不参与 plan.summary，UI 仅用于提示"目标端存在未受管 Skill"。

## 3. 验证机制

### 自动化测试

- 新增集成测试 `tests/integration/scan.test.ts`，覆盖：
  - 目标目录不存在：`scan` 不抛错，输出"未检测到"；
  - 目标端同名 Skill 且 checksum 相同：`identical`；
  - 目标端同名 Skill 且 checksum 不同、无 deploy tag：`conflict`；
  - 目标端同名 Skill 且 checksum 不同、有 deploy tag：`changed`；
  - 目标端存在未受管 Skill：`untracked`；
  - `diffDirectories` 对 added/removed/changed 三类文件均能识别；
- 执行 `pnpm run typecheck`；
- 执行 `pnpm run test`。

### 手动检查

- 运行 `pnpm dev list` 检查列表输出；
- 运行 `pnpm dev scan` 检查扫描状态输出（含 untracked）；
- 运行 `pnpm dev diff <skill> claude:user` 检查文件比对差异输出。

## 4. 风险与待确认

- **deploy 标记的写入语义**：D3a 才会补 `writeDeployTag` 的覆盖语义；若 D2 阶段读到 `sourcePath` 与 registry 中 `localPath` 不一致的 deploy tag（例如被搬移），目前只展示，不修改 registry。
- **Gemini 目标根目录**：当前 `config.targets.gemini.userSkillPath` 默认指向 `~/.gemini/config/skills`，D2 只扫描不写入；与 D3a 范围一致。
- **`mtime` vs checksum**：D2 仅以 checksum 判定状态，避免复制时间造成的 mtime 漂移；多向同步的 mtime 比对留给 D3b 之后。
