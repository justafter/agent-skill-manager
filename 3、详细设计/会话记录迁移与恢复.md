# 会话记录迁移与恢复详细设计

本文档描述 D11 会话管理模块的核心接口、状态机和安全边界。任务拆分见 [`4、计划/D11会话记录迁移与恢复开发计划.md`](../4、计划/D11会话记录迁移与恢复开发计划.md)。

## 1. 领域边界

会话记录与 Skill 安装是两个独立领域：

- `src/adapters` 继续只负责 Skill 目录扫描。
- `src/sessions/adapters` 负责会话记录识别和源路径映射。
- 文件迁移和还原统一由 `src/sessions/apply.ts` 执行。
- 路由和 UI 不持有任何复制、删除逻辑。

## 2. 状态机

```text
planned
  -> copying
  -> verified
  -> committed
  -> completed
                 \
                  -> cleanupPending

任意提交前失败 -> failed（删除 staging，保留源）
提交后删除失败 -> cleanupPending（源和归档都保留）
```

还原使用相同状态机，方向相反。`committed` 表示目标端已经存在通过 checksum 校验的完整副本。

## 3. 活动状态

`activity` 统一输出：

- `idle`：已通过适配器活动标记、稳定窗口和锁探测。
- `busy`：命中活动 sessionId、文件持续变化或平台锁冲突。
- `unknown`：当前平台不能可靠判断。

首版只有 `idle` 可以 apply。`busy` 和 `unknown` 都需要用户关闭对应 Agent 后重新扫描、重新生成 plan。

## 4. 路径安全

- 迁移源必须位于适配器声明的精确源根内。
- 迁移目标必须位于 `<archiveDir>/sessions/<agent>/<sessionId>` 内。
- 还原目标通过当前配置源根与 manifest 的安全相对路径重新计算。
- `originalPath` 不参与路径决策。
- 所有现存父目录先 realpath，再进行 `path.relative` 越界判断。
- 拒绝 manifest 中的绝对相对路径、`..`、空路径和非 UUID sessionId。
- archiveDir 与源根不得相互包含。

## 5. 完整性

文件 checksum 使用 SHA-256；目录 checksum 由排序后的相对路径、文件类型和文件内容 hash 共同计算。manifest 不参与 payload checksum。

每次迁移校验：

- 文件总数。
- 逻辑字节总数。
- payload checksum。
- plan 和 apply 时的源文件统计及 checksum。

复制后同时重新计算源和 staging checksum。只有两者一致且源快照未发生漂移，才能提交归档并删除源。

## 6. 冲突

- 迁移目标 bundle 已存在：`ARCHIVE_CONFLICT`。
- 还原目标已存在：`RESTORE_CONFLICT`。
- manifest 与 payload 不一致：`INTEGRITY_CHECK_FAILED`。
- plan 后源发生变化：`SOURCE_CHANGED`。
- 活动或锁定：`SESSION_BUSY`。
- 空闲状态未知：`SESSION_ACTIVITY_UNKNOWN`。

首版不提供绕过冲突或强制覆盖的开关。

## 7. 可恢复操作日志

操作日志写入：

```text
<archiveDir>/.agent-skill-manager/operations/<operationId>.json
```

每次状态变化使用原子写入，内容包括：

- 操作方向和 planId。
- Agent、sessionId、源、目标。
- 当前状态和时间。
- manifest/checksum 摘要。
- 失败 code、message 和可恢复动作。

首版通过 `GET /api/sessions/operations` 读取持久化状态。自动清理中断 staging、重试 `cleanupPending` 需要再次判断哪一侧是完整权威副本，留到恢复向导实现；首版不自动删除现场数据。

## 8. UI 统计口径

- `agentBytes`：当前 Agent 会话源目录中的逻辑字节。
- `archiveBytes`：有效 manifest 对应 payload 的逻辑字节。
- `migratedBytes`：当前有效归档中已从源端删除的逻辑字节。
- `selectedBytes`：当前勾选项逻辑字节。

不把逻辑字节描述成文件系统实际释放块数。

## 9. 展示交互

会话页参考 [cc-switch Session Manager](https://github.com/farion1231/cc-switch/blob/main/src/components/sessions/SessionManagerPage.tsx) 的紧凑列表和主从详情信息层级，但不复制其业务边界：

- Agent 目录和归档目录继续同时展示，保证迁移方向直观可见。
- 列表行展示 Agent 标识、两行标题、相对更新时间、大小、项目简称和状态点；复选框只负责批量选择，点击正文只切换详情。
- 工具栏支持“列表 / 分类”切换；分类视图按“当前 Agent → `workspacePath` 项目目录”折叠展示，并显示 Agent、项目和记录数量。
- 点击会话后，详情侧按需请求 transcript，展示用户、助手、系统提示和工具消息；元信息、迁移状态与单条 plan 入口放在同一详情侧。
- 宽屏为“双列表 + 详情”三栏；中等宽度详情下沉；窄屏列表改为单列并保持独立滚动。
- 列表/分类偏好保存在浏览器本地；搜索时分类组自动展开，避免命中项被折叠隐藏。
- 不提供 cc-switch 的永久删除、复制恢复命令或启动终端能力。

## 10. 对话读取边界

- API：`GET /api/sessions/:agent/:location/:sessionId/messages`，只允许读取扫描结果中已识别的 UUID 会话。
- Agent 目录记录复用 `assertSessionEntriesSafe`；归档记录必须先通过 manifest 和归档根路径校验。Gemini transcript 还需验证真实路径仍位于对应 UUID 目录内。
- Claude 解析 `user` / `assistant` / `system` 和 text、tool_use、tool_result；跳过 thinking。
- Codex 只解析 `response_item` 中的 user、assistant、system 和工具调用/结果；跳过 developer 指令与 reasoning，避免展示内部提示和重复事件。
- Gemini 解析 `USER_INPUT`、`PLANNER_RESPONSE`、系统消息与工具事件；跳过会重复历史的 `CONVERSATION_HISTORY` 和 thinking。
- 单次读取最多处理 100,000 行、1,000 条消息、2,000,000 个正文字符，单条消息最多 40,000 字符；达到上限时返回 `truncated` 与可展示 warning。
- 对话只在选中会话后按需读取；读取接口无写副作用，Agent 原目录和归档 payload 均不被修改。
