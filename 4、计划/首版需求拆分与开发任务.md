# 首版需求拆分与开发任务

日期：2026-07-03

## 1. 当前基线

本仓库已经具备首版的目录、类型和入口骨架，但多数功能仍处于可编译骨架或占位状态。

已具备：

- 文档分层：`1、需求`、`2、架构`、`3、详细设计`、`4、计划`、`5、开发`。
- 技术栈骨架：TypeScript、Commander、Express、React + Vite。
- 基础类型：Skill、Adapter、Plan、Project、Rule、Backup。
- 基础能力雏形：
  - 配置加载：`src/core/config.ts`
  - Skill 校验：`src/validation/skill.ts`
  - 目录 hash / mtime：`src/utils/hash.ts`、`src/validation/mtime.ts`
  - Plan 内存注册：`src/core/plan.ts`、`src/core/state.ts`
  - 项目扫描：`src/projects/scanner.ts`
  - 托管块替换：`src/rules/block.ts`
  - 备份索引创建：`src/backup/create.ts`

仍需开发：

- 注册表读写与导入落库。
- scan/list/sync/backup/restore/watch 等 CLI 主流程。
- sync plan/apply 的真实文件复制、备份、冲突判断、管理标记。
- server API 与 CLI 共用核心逻辑。
- Web UI 从真实 API 拉取数据并执行 plan/apply。
- 项目级 Skill 注入与规则模板同步。
- Antigravity 加载验证与 Watch Mode。

## 2. 首版边界

首版必须完成：

- 本地 Skill 导入、校验、注册、列表展示。
- Claude / Codex 用户级 Skill 扫描与同步。
- dry-run plan 与 apply 严格分离。
- 同步前备份、冲突检测、管理标记写入。
- 项目注册、项目扫描、项目级 Skill 注入。
- `CLAUDE.md` / `AGENTS.md` / `GEMINI.md` 规则模板 diff、托管块更新、拉取为模板。
- CLI 与 Web UI 共享同一套核心能力和数据文件。

首版暂不做：

- 远程市场、远程搜索、联网自动更新。
- ZIP 安装。
- OpenCode / Hermes 写入适配。
- Cursor / Windsurf 规则自动写入。
- 多模板复杂合并。
- 系统托盘、安装包、多用户权限。

## 3. 开发顺序建议

优先走 CLI 到核心闭环，再补 API 和 Web UI。原因是文件系统同步、备份、冲突检测是高风险逻辑，应先在终端和测试中跑通，再接入界面。

推荐顺序：

1. D0 基础设施补齐。
2. D1 Skill 导入与注册表。
3. D2 用户级 scan/list/diff。
4. D3 用户级 sync plan/apply/backup/tag。
5. D4 备份恢复闭环。
6. D5 server API 复用核心能力。
7. D6 Web UI 接真实数据。
8. D7 项目工作区与项目级注入。
9. D8 规则模板同步。
10. D9 Watch Mode 与 Antigravity 验证。

## 4. 任务拆分

### D0 基础设施补齐

目标：先把配置、注册表、路径安全和错误模型整理成可复用基础。

开发项：

- 新增 registry 读写模块，统一读写 `library/registry.json`。
- 明确用户级配置与仓库默认配置的合并策略：
  - 仓库默认配置 `skill-manager.config.json` 作为只读基准配置。
  - 用户自定义配置（如注册项目、激活的适配器配置）统一落库到用户目录的 `%USERPROFILE%\.skill-manager\config.json` 中，并在运行时与仓库默认配置深度合并（Deep Merge）。
  - 若当前代码或旧文档仍使用 `%USERPROFILE%\.skill-manager.config.json`，D0 阶段需要一并迁移到目录式用户配置路径，避免后续 PID、日志、配置文件落点分裂。
- 补齐路径工具与安全校验机制（Path Guard）：
  - 补齐路径工具：绝对路径解析、`%USERPROFILE%` 展开。
  - 实现路径越界校验（Path Guard）：支持安全路径校验，确保任何写入操作只能限制在已注册的项目目录或特定的目标 Agent 技能目录内部，拒绝包含 `..` 或未解析符号链接的越界路径。
  - 白名单只允许具体写入根：`library/skills`、`library/registry.json`、`backups`、`ruleTemplateDir`、已注册项目目录、已启用适配器的用户级 Skill 目录；不得把整个仓库根目录作为通用安全写入根。
- 引入统一错误类型，至少包含 `code`、`message`、`details`。
- 补充核心基础测试。

涉及文件：

- `src/core/config.ts`
- `src/core/engine.ts`
- `src/utils/paths.ts`
- `src/utils/fs.ts`
- `src/projects/guard.ts`
- `library/registry.json`

验收标准：

- `asm doctor` 能输出配置、目标路径存在性、可写性。
- registry 读写有测试覆盖。
- 越界路径会被拒绝，并能返回明确错误。

### D1 Skill 导入与注册表

目标：让 `asm import <path>` 真正把 Skill 纳入管理器。

开发项：

- 校验 `SKILL.md` frontmatter：`name`、`description`、目录名一致。
- 计算 checksum、lastModified、资源目录标记。
- 把导入结果写入 `library/registry.json`。
- 导入时复制一份到 `library/skills/<skill-name>`，并在注册表中保留原始 `localPath` 作为外部开发目录引用。
- 首版同步、扫描和目标端写入默认以 `library/skills/<skill-name>` 作为 canonical source；`localPath` 只用于溯源、重新导入、后续 Watch Mode 和开发目录变更提醒。
- 重复导入时给出覆盖、跳过或更新策略；覆盖 `library/skills/<skill-name>` 前必须备份旧库目录和 registry snapshot，避免 force copy 后残留旧文件或 registry 与库目录不一致。

涉及文件：

- `src/cli/import.ts`
- `src/validation/*`
- `src/sync/copy.ts`
- `src/core/fingerprint.ts`
- `library/registry.json`

验收标准：

- 导入合法 Skill 后，`registry.json` 出现对应记录。
- 缺少 `SKILL.md`、缺少字段、目录名不一致时导入失败。
- 重复导入不会产生脏数据。

### D2 用户级扫描、列表与差异

目标：能看到本地库和 Agent 端的真实状态。

开发项：

- 实现 `asm list`，展示 registry 中所有 Skill。
- 实现 `asm scan`，扫描本地库、Claude 用户级目录、Codex 用户级目录、Gemini 目标目录。
- 适配器补齐 `scanUserSkills()`。
- 实现 `asm diff <skill-name> <target>`，按 registry 源与目标端对比。
- 扫描结果能识别 identical、missing、changed、conflict。

涉及文件：

- `src/cli/list.ts`
- `src/cli/scan.ts`
- `src/cli/diff.ts`
- `src/adapters/*`
- `src/sync/multidirection.ts`
- `src/rules/diff.ts`

验收标准：

- 无目标目录时给出未检测到，而不是异常退出。
- 目标端存在同名 Skill 时能计算状态。
- diff 输出能定位文件级和文本级差异。

### D3 用户级同步 plan/apply

目标：跑通最小高价值闭环：本地 Skill 从 `library/skills` 同步到 Claude / Codex 用户级目录。

开发项：

- 扩展 plan 输入：`skillName`、`from`、`targets`。
  - D3a 只支持推送同步：`from: local -> targets: claude:user/codex:user`，其中 `local` 明确指 `library/skills/<skill-name>`。
  - D3b 后置到 D4 之后：再实现反向拉取和跨 Agent 多向同步，避免在备份恢复闭环未完成前覆盖本地库。
- plan 阶段读取源端和目标端 checksum，生成 create / modify / skip / conflict。
- apply 阶段执行备份、复制、写 `.skill-manager-deploy.json`。
- 冲突目标默认不写入。
- apply 成功后更新 registry 的 `syncedTargets`。
- CLI 支持：
  - 推送同步：`asm sync <skill-name> --to claude:user,codex:user [--dry-run]`
  - 反向拉取命令暂不进入 D3a；后续 D3b / 多向同步阶段再补：`asm sync <skill-name> --from claude:user --to local [--dry-run]`

涉及文件：

- `src/cli/sync.ts`
- `src/core/plan.ts`
- `src/core/apply.ts`
- `src/core/conflict.ts`
- `src/sync/engine.ts`
- `src/sync/copy.ts`
- `src/sync/tag.ts`
- `src/adapters/claude.ts`
- `src/adapters/codex.ts`

验收标准：

- dry-run 不产生任何文件变更。
- apply 前创建备份。
- hash 相同目标自动 skip。
- 非本工具管理的不同内容目标标记 conflict，不覆盖。
- 管理标记存在且 hash 不同时允许生成 modify。
- D3a 不覆盖 `library/skills`，不修改外部 `localPath`。

### D4 备份与恢复

目标：覆盖写入前可回滚，恢复流程能独立验收。

开发项：

- 创建备份目录和 `index.json`。
- 备份 Skill 目录、规则文件、registry snapshot。
- 实现 `asm backup` 和 `asm restore <backup-id>`。
- 实现备份列表 API / CLI 输出。
- 增加备份清理策略入口，首版可先只提供手动清理或保留配置。

涉及文件：

- `src/backup/*`
- `src/cli/backup.ts`
- `src/cli/restore.ts`
- `src/server/routes/backup.ts`
- `src/server/routes/restore.ts`

验收标准：

- 覆盖前自动生成备份索引。
- restore 能把目标文件恢复到备份时状态。
- registry 可恢复到写入前快照。

### D5 Server API

目标：让 Web UI 和 CLI 使用同一套核心能力，不在路由中重复写逻辑。

开发项：

- `GET /api/skills` 返回 registry + scan 状态。
- `POST /api/import` 调用导入核心能力。
- `POST /api/scan` 调用扫描核心能力。
- `POST /api/sync/plan` 生成 plan。
- `POST /api/sync/apply` 只接受 `planId`。
- `GET /api/diff` 返回文本 diff。
- 所有写接口做 zod 入参校验。

涉及文件：

- `src/server/app.ts`
- `src/server/routes/*`
- `src/core/engine.ts`
- `src/types/*`

验收标准：

- API 与 CLI 对同一个 Skill 的 plan summary 一致。
- apply 不允许携带临时目标绕过 plan。
- 错误响应包含可展示的错误 code 和 message。

### D6 Web UI 首版闭环

目标：Web 页面能真实管理 Skill，而不是展示占位行。

开发项：

- Skill 列表接入 `GET /api/skills`。
- 添加状态过滤：全部、缺失、已同步、冲突、项目级。
- 导入页面接入 `POST /api/import`。
- 同步按钮先创建 plan，再弹确认窗口。
- `PlanConfirmDialog` 展示 create / modify / skip / conflict 统计和路径。
- 备份页展示备份列表并支持恢复确认。
- 所有写操作成功后刷新 query cache。

涉及文件：

- `web/src/api/client.ts`
- `web/src/pages/*`
- `web/src/components/*`
- `web/src/hooks/*`

验收标准：

- 页面不再出现固定的 `example-skill` 占位数据。
- 同步前必须出现确认窗口。
- 冲突项在 UI 中明确标红并默认不可 apply。
- 备份恢复有二次确认。

### D7 项目工作区与项目级 Skill 注入

目标：实现按项目隔离安装 Skill。

开发项：

- 实现 `asm project list/add/scan`。
- 项目配置统一写入用户级配置 `%USERPROFILE%\.skill-manager\config.json`，仓库默认配置只提供默认结构和示例，不写入用户项目清单。
- 实现 `asm project inject <project-id> <skill-name> --agent <agent> --dry-run`。
- 项目级写入路径必须落在注册项目内部。
- 项目级安装状态写入 registry 的 `projectInstalls`。
- Web 项目空间页面接入真实数据。

涉及文件：

- `src/cli/project.ts`
- `src/projects/*`
- `src/adapters/*`
- `src/server/routes/projects.ts`
- `web/src/pages/ProjectSpacePage.tsx`
- `web/src/components/ProjectList.tsx`

验收标准：

- 未注册项目不能注入。
- 项目路径越界被拒绝。
- 能注入到 `.claude/skills/<skill-name>` 和 `.agents/skills/<skill-name>`。
- 注入前有 dry-run，覆盖前有备份。

### D8 规则模板同步

目标：支持项目 `CLAUDE.md` / `AGENTS.md` / `GEMINI.md` 的安全更新和回收。

开发项：

- 读取 `library/rules/<agent>/<file>` 模板。
- 实现规则 plan：create、overwrite、block、pull-template。
- 托管块模式支持新增块和替换块。
- 无托管块的已有文件默认只出 diff，不自动覆盖。
- 实现 CLI：`plan-rules`、`push-rules`、`pull-rules`。
- Web 项目详情页提供规则 diff 与写入模式选择。

涉及文件：

- `src/rules/*`
- `src/cli/project.ts`
- `src/server/routes/projects.ts`
- `web/src/components/RuleTemplateEditor.tsx`
- `web/src/components/DiffView.tsx`

验收标准：

- block 模式保留非托管内容。
- overwrite 模式必须显式确认。
- pull-template 不覆盖已有模板，除非明确传入覆盖选项。
- `CLAUDE.md`、`AGENTS.md`、`GEMINI.md` 都有测试覆盖。

### D9 Watch Mode 与 Antigravity

目标：补齐首版增强能力，但排在安全同步闭环之后。

开发项：

- `asm watch <skill-name> --target <target>` 启动 chokidar 监听。
- 只监听用户显式开启的 Skill。
- 防抖后自动生成 plan 并 apply 到已确认目标。
- 失败写入 watch error log。
- Antigravity 适配器生成 `plugin.json`，写入插件 skills 目录。
- UI 显示 Antigravity 状态：路径已写入、加载已验证、加载未验证。

涉及文件：

- `src/cli/watch.ts`
- `src/server/routes/watch.ts`
- `src/adapters/gemini-antigravity.ts`
- `src/utils/pidfile.ts`

验收标准：

- Watch 默认关闭。
- 修改 Skill 后能同步到已配置目标。
- Watch 同步失败不会中断进程。
- Antigravity 无法实机确认时，状态必须显示“加载未验证”。

## 5. 第一轮开发建议

第一轮只做 D0 到 D3，形成最小可用闭环：

```text
导入 Skill
  -> 写入 registry
  -> list / scan 可见
  -> sync --dry-run 生成真实 plan
  -> sync apply 写入 Claude / Codex 用户级目录
  -> 写入前备份
  -> 目标端生成管理标记
```

第一轮只包含 D3a 推送同步，不包含反向拉取、跨 Agent 同步、Web UI 和项目级规则文件。这样可以先把高风险的文件写入逻辑打稳，后续 Web 只是调用同一套 API。

## 6. 测试拆分

必须优先补的测试：

- `parseSkillDir`：合法、缺 `SKILL.md`、缺字段、目录名不一致。
- registry：读空表、写入、更新、重复导入。
- checksum：相同内容 hash 相同，内容变更 hash 变更。
- plan：不存在目标 create，相同目标 skip，不同且无标记 conflict，不同且有标记 modify。
- apply：dry-run 无写入，apply 有备份和管理标记。
- restore：恢复目录和 registry snapshot。
- rules block：新增托管块、替换托管块、保留非托管内容。
- project guard：拒绝写出项目目录。

集成测试优先场景：

1. 临时 Skill 导入到临时 registry。
2. 同步到临时 Claude 用户级目录。
3. 再次同步命中 skip。
4. 手动改目标端内容后识别 conflict。
5. 写入管理标记后识别 modify。
6. restore 回到覆盖前状态。

## 7. 开发验收口径

每个任务完成时至少满足：

- CLI 命令可执行，并有清晰输出。
- 关键分支有自动化测试。
- 写文件前能说明目标路径。
- dry-run 和 apply 结果一致。
- 失败时不留下半写入状态，或有备份可恢复。
- 文档中对应命令和行为同步更新。
