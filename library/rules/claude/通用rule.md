## 通用工作规则

- 修改前先理解现有结构、命名和风格。
- 优先使用项目已有模式，不随意引入新框架、新依赖或大规模重构。
- 只修改与当前任务直接相关的文件。
- 如果发现与任务无关的问题，先记录，不要顺手扩大修改范围。
- 涉及风险、破坏性操作、数据迁移、权限变更时，必须先说明并等待确认。

## 工作留痕规则

只在实际修改项目文件时创建或更新 `AIWorkLog/` 工作日志。需求开发和 Bug 修复必须在开始修改前创建或更新日志；简单修改可以在完成后补充一条简短记录。只读分析、解释代码、搜索文件、方案讨论或未实际修改项目文件时，不需要创建 `AIWorkLog/`。

### 任务类型

- 简单修改：不改变业务逻辑、不新增用户可见能力、不修复缺陷的文案调整、配置调整、小范围样式修改、小范围重构等。
- 新增功能 / 需求开发：新增能力、功能改造、流程变化、用户行为变化，或新增页面、接口、模块、自动化能力等。
- Bug 修复：已有能力与预期不一致、报错、异常、回归问题，或修复已有功能的行为缺陷。

### 日志目录

- `AIWorkLog/BugFixLog/`：记录 bug 修复。
- `AIWorkLog/FeatureDevelopmentLog/`：记录新增功能、需求开发。
- `AIWorkLog/SimpleChangeLog/`：记录简单修改、配置调整、文案调整、小范围重构等。

### 文件规则

- 每个问题、需求或修改事项必须单独创建一个 Markdown 文件。
- 文件命名建议使用：`YYYY-MM-DD-任务简述.md`。
- 示例：
  - `AIWorkLog/BugFixLog/2026-06-26-登录按钮无响应.md`
  - `AIWorkLog/FeatureDevelopmentLog/2026-06-26-新增导出功能.md`
  - `AIWorkLog/SimpleChangeLog/2026-06-26-调整首页文案.md`

### 持续记录要求

- 纯查询、纯分析、方案讨论、解释代码、搜索文件、未实际修改项目文件时，不创建 `AIWorkLog/`。
- 不记录每个命令、每个文件读取、每个中间推理。
- 只记录对后续 agent 有帮助的关键事实。
- 简单任务可以只保留一次完整的实施与验证记录，不需要机械拆成多轮步骤。
- 复杂任务也不要逐步骤记录流水账，只记录关键问题、关键决策、方案变化、阻塞点、重要验证和最终结论。
- 遇到阻塞、方案变化、上下文可能中断、长任务、多阶段开发时，只追加关键记录，不要记录每个执行步骤。
- 日志目标是让后续 agent 快速理解当前状态，不是逐条复述每个命令步骤。
- 如果上下文、额度或会话突然中断，后续 agent 应能通过日志知道当前在做什么、已经做了什么、修改了哪些文件、验证到了哪一步、接下来该怎么做，以及哪些尝试不要重复。

### 接手已有任务

- 接手已有任务时，必须先阅读对应的 `AIWorkLog/` 文件。
- 阅读后再查看当前代码状态和相关文件。
- 继续记录时只追加关键变化、当前结论和下一步，不重复整理已有记录。

### 需求开发额外记录

新增功能、需求开发或功能改造必须先记录需求是什么、边界是什么、验收标准是什么。需求开发日志必须存放在 `AIWorkLog/FeatureDevelopmentLog/`，并且每个需求单独一个 Markdown 文件。

- 简单需求可以只写一次完整的“需求说明 + 实现方案 + 验收结果”。
- 复杂需求只追加关键问题、关键决策、范围变化、阻塞点和验收结论。
- 如果需求范围发生变化，必须更新 `范围边界` 和 `验收标准`。
- 如果另一个 agent 接手，应能通过该记录判断需求原始目标、已实现内容、不做内容、当前验证结果和下一步。

## 需求说明

- 验证方式：

### Bug 修复额外记录

每次修复 bug 时，必须先记录 bug 判断、关键修复点、失败尝试原因、最终有效方案和验证结论。Bug 修复日志必须存放在 `AIWorkLog/BugFixLog/`，并且每个 bug 单独一个 Markdown 文件。多次尝试只记录关键失败原因，不记录每个执行步骤。

```md
## Bug 描述

- 触发条件：
- 预期结果：
- 实际结果：
- 影响范围：
- 初步判断：
```

## 验证规则

- 修复 bug 后必须运行与修改范围匹配的测试、检查或手动验证。
- 如果无法验证，必须明确说明原因和未验证风险。
- 不要声称“已修复”，除非已经完成合理验证。

## 输出要求

- 回复要简洁、具体、可执行。
- 说明修改了什么、为什么这样修、如何验证。
- 对不确定内容要标明假设，不要把推测当事实。
