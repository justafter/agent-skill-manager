## 2. 运行环境与启动模型

### 2.1 三种运行模式

```text
┌────────────────────────────────────────────────────────────┐
│  模式 1: CLI-only                                          │
│  $ asm <subcommand>                                        │
│  → 直接调用 src/core/* 内的函数                            │
│                                                            │
│  模式 2: 本地 server + CLI 调用                            │
│  $ asm server start                                        │
│  → 起 Express on 127.0.0.1:47821                          │
│  $ asm sync ...                                            │
│  → CLI 通过 HTTP 调本机 server，保持 plan/apply 单一路径   │
│                                                            │
│  模式 3: Web UI                                            │
│  $ asm web                                                 │
│  → vite dev (前端) + 同一个本地 server (后端)              │
└────────────────────────────────────────────────────────────┘
```

**首版实现**：

- `asm <subcommand>` 直接调核心函数，不强制起 server。
- `asm web` 启动 vite + 同进程 server。
- `asm server start` 用于让多个 CLI 实例复用同一个 plan/apply 注册表。

### 2.2 配置文件

- `skill-manager.config.json`（仓库根）— 仓库级默认。
- `~/.skill-manager/config.json`（用户级）— 首次启动时由默认配置生成，覆盖仓库级。
- 加载顺序：CLI flags > env > 用户级 > 仓库级。代码内部统一使用合并后的 `ResolvedConfig`，避免各模块直接判断配置来源。

### 2.3 进程模型

- `asm` CLI 一次性进程。
- `asm web` 同时拉起 vite + express（单进程 two-server）。
- 通过 `~/.skill-manager/server.pid` 防止多实例。
- Watch Mode 通过 IPC 信号与 server 通信（每个 Skill 一个 watcher）。

**约束**：

- `core/` 不依赖 `adapters/`：核心引擎只通过 `Adapter` 接口工作。
- `adapters/` 不直接决定冲突策略：策略由 `core/conflict.ts` 决定。
- `cli/` 和 `server/` 是同一套核心 API 的两个入口，**禁止在 CLI 单独实现写逻辑**。
- `validation/` 是纯函数，不持有 IO 状态。

## 4. 关键技术决策

### 4.1 Plan / Apply 分离

**问题**：用户希望在 UI 上看到 diff 后再写入，且执行内容必须与展示内容严格一致。

**方案**：

```ts
type Plan = {
  planId: string;
  createdAt: string;
  source: string;
  items: PlanItem[];
  backupId?: string;       // plan 阶段预分配；apply 成功后成为实际备份索引
};

type PlanItem =
  | { kind: 'create'; target: string; bytes: number }
  | { kind: 'modify'; target: string; checksumBefore: string; checksumAfter: string }
  | { kind: 'skip';   target: string; reason: 'identical' }
  | { kind: 'conflict'; target: string; managedBy?: string; checksumBefore: string; checksumAfter: string }
  | { kind: 'delete'; target: string };
```

- `core/plan.ts` 暴露 `createPlan(intent): Plan`。
- `core/apply.ts` 暴露 `applyPlan(planId): ApplyResult`，要求 `planId` 必须存在于 `core/state.ts` 的内存表（默认 TTL 15 分钟）。
- Server 端以请求体中的 `planId` 为准校验一致性；如后续需要幂等重试，可额外接受 `Plan-Id` header，但不能覆盖请求体。

### 4.2 完全复制 vs 符号链接

**结论**：首版使用 `fs.cp` 完全复制。

- **优势**：跨分区、备份工具、IDE 索引都对纯文件最友好。
- **劣势**：磁盘占用略大。
- **保留扩展口**：`sync/copy.ts` 抽象为 `Copier` 接口，后续可加 `SymlinkCopier`。

### 4.3 SHA-256 目录 checksum 与 mtime

**方案**：

```
checksum = sha256(
  排序后拼接 (相对路径 + "\0" + sha256(文件内容))
)

lastModified = max(目录内文件 mtime)
```

- 排序保证确定性。
- `checksum` 只由相对路径和文件内容决定，用于同步跳过、覆盖和冲突判断。
- `lastModified` 单独记录，仅用于多端扫描时推断最新来源，不参与 `checksum`。
- 文件内容或相对路径变化 → `checksum` 变化；仅复制导致的 mtime 变化不应触发内容冲突。

### 4.4 冲突检测

**三种状态**：

| target 端状态 | 策略 |
| --- | --- |
| 不存在 | `create` |
| 存在，checksum 相同 | `skip`（identical） |
| 存在，checksum 不同，**有** `.skill-manager-deploy.json` | `modify`（可覆盖，先备份） |
| 存在，checksum 不同，**无** 管理标记 | `conflict`（暂停，需用户显式 confirm） |

### 4.5 托管块格式

```md
<!-- BEGIN AgentSkillManager:<agent-id> -->
... 由管理器维护的规则片段 ...
<!-- END AgentSkillManager:<agent-id> -->
```

- 解析：`src/rules/block.ts` 用正则定位 `BEGIN/END`。
- 替换：保留行外内容，更新块内。
- 缺失：可选追加（在文件末尾或指定锚点）。

### 4.6 Watch Mode

```ts
const watcher = chokidar.watch(localPath, {
  ignored: /(^|[\\\/])\../,         // 忽略隐藏文件
  persistent: true,
  ignoreInitial: true,
  awaitWriteFinish: { stabilityThreshold: 300, pollInterval: 100 },
});

watcher.on('change', async (file) => {
  await debouncedSync(skill, target);
});
```

- 每个被开启 Watch 的 Skill 一个 watcher。
- `debouncedSync` 300ms 合并多次写。
- 失败写入 `~/.skill-manager/watch-error.log`，不中断监听。

### 4.7 备份索引与清理

- 每次 `create` 写一个 `index.json` + 实际文件。
- 清理策略：保留最近 50 次备份 + 30 天内所有备份，按 OR 取并集。
- 备份目录体积超过 2 GB → 触发清理。

## 5. 数据契约

### 5.1 主要类型

```ts
// src/types/skill.ts
export type Scope = 'user' | 'project';
export type TargetKey = `${'claude' | 'codex' | 'gemini'}:${Scope}`;

export interface SkillMeta {
  name: string;            // 唯一标识
  version: string;         // semver
  description: string;     // 必填
  localPath: string;       // 源目录绝对路径
  checksum: `sha256:${string}`;
  hasScripts: boolean;
  hasReferences: boolean;
  hasAssets: boolean;
  lastModified: string;    // ISO
}

export interface SkillState extends SkillMeta {
  syncedTargets: TargetKey[];
  projectInstalls: {
    projectId: string;
    target: TargetKey;
    checksum: `sha256:${string}`;
    deployedAt: string;
  }[];
}
```

### 5.2 API 契约（部分）

```http

Content-Type: application/json

  "skillName": "comfyui-video-gen",

```

```http

```

## 6. 安全模型

| 风险 | 缓解 |
| --- | --- |
| 越界写入项目目录 | `projects/guard.ts` 在 `realpath` 后做前缀判断 |
| 越界写入 Agent 目录 | `adapters/base.ts` 校验目标路径在 `userSkillPath` / `projectSkillPath` 内 |
| 覆盖未管理 Skill | 写入前读 `.skill-manager-deploy.json`，不存在则 `conflict` |
| 误删文件 | 所有删除操作必须显式 confirm，CLI 默认 dry-run |
| Watch 自动覆盖 | 默认关闭；按 Skill 显式开启 |
| 进程冲突 | PID 文件 + 端口占用检测 |

## 7. 性能与扩展性

| 维度 | 目标 | 措施 |
| --- | --- | --- |
| 扫描速度 | 1k Skill 全量扫描 < 5s | 目录 checksum 缓存 + 并行 IO |
| 同步单 Skill | < 1s（无 diff） | 只读 fingerprint，不复制未变更 |
| 同步批 50 Skills | < 30s | 并行（pool = 4） |
| 启动时间 | < 500ms | 懒加载适配器 |
| 内存 | < 200MB | plan/apply 用流式处理 |

## 8. 测试策略

- **单元**：`src/**/*.test.ts`，覆盖 `validation/`、`core/conflict.ts`、`rules/block.ts`。
- **集成**：在 `tests/integration/` 用 `node:test` 跑端到端：
  - 创建临时 Skill → import → sync → 检查目标端 → 备份 → 恢复。
  - 创建临时项目 → 注册 → 注入 Skill → push 规则（block）→ 检查。
- **E2E（首版可选）**：`tests/e2e/` 通过 `playwright` 启 Web UI + server，验证关键页面。
- **覆盖率门槛**：核心模块（`core/`、`validation/`、`rules/block.ts`、`sync/`）≥ 80%。

## 9. 监控与日志

- CLI：`pino` 漂亮输出，level 由 `--verbose` 控制。
- Server：JSON 行日志到 `~/.skill-manager/server.log`。
- Watch：error log 到 `~/.skill-manager/watch-error.log`。
- 关键事件：`plan.created`、`plan.applied`、`plan.conflict`、`backup.created`、`backup.restored`、`watch.error`。

## 10. 部署与分发

- **首版**：本地 `git clone` + `pnpm install` + `pnpm run build`。
- **包形态**：可选发布到 npm 作为全局 `asm` 命令（`bin.asm = "./dist/cli/index.js"`）。
- **不引入**：自动更新器、安装包打包（NSIS / msi）、系统托盘。

## 11. 风险与对策

| 风险 | 等级 | 对策 |
| --- | --- | --- |
| Antigravity 自定义插件目录加载失败 | 中 | 首版写入并标记"加载未验证"，由 UI 显示 |
| Codex 路径规范变化 | 中 | 适配器只读一份 `paths.config.ts`，方便后续调整 |
| 大目录 checksum 计算慢 | 低 | 并行 + 缓存 |
| 多端并发覆盖 | 低 | 默认 Watch 只推送，不从外部拉取 |
| 备份膨胀 | 低 | 自动清理 + UI 显示占用 |

## 12. 开发约定

- **代码风格**：2 空格缩进，单引号，无分号（与 Node 社区主流一致），TypeScript strict。
- **命名**：文件 `kebab-case.ts`，类 `PascalCase`，变量/函数 `camelCase`，类型 `PascalCase`。
- **错误**：使用自定义 `AsmError` 类 + code 字段，CLI 层负责转人类可读。
- **日志**：使用 `pino` 子 logger，不在核心库 `console.log`。
- **提交**：Conventional Commits（`feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:`）。
- **分支**：`main` 受保护；功能在 `feat/<scope>`，完成后 squash merge。
